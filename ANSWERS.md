# Answers

## 1.

The current `OrdersService` stores orders in a module-level `const ORDERS: Order[]` array defined in `orders.service.ts`. This works for a demo but breaks immediately under production conditions: data is lost on every restart, concurrent writes to the array are not safe, and there is no way to query, paginate, or index the data efficiently.

For a production QDC system I would introduce a proper persistence layer in three steps. First, replace the in-memory array with a database — PostgreSQL is the natural fit given the relational structure (one Order has many Garments, each with its own status). An ORM like TypeORM or Prisma would let me model `Order` and `Garment` as entities with a one-to-many foreign key relationship, eliminating the duplication of types that currently exists between `server/src/orders/orders.service.ts` and `client/src/App.tsx`. Second, the service methods `findAll()` and `findOne(id)` would become async and delegate to a repository, enabling the database to handle filtering, pagination (`LIMIT`/`OFFSET` or cursor-based), and indexed lookups by `id` or `customerName` instead of a full array scan. Third, at scale I would add a read-through cache (e.g., Redis) in front of `findAll()` for the list view, since that endpoint is hit on every dashboard load and the data changes less frequently than it is read. On the write path, an event bus or job queue (e.g., BullMQ) would decouple status update processing from the HTTP request cycle, which matters once garment status changes trigger side effects like SMS notifications or billing events.

## 2.

The current `getOrder` handler in `orders.controller.ts` returns `Order | { error: string }` with an HTTP `200 OK` regardless of whether the order was found. This is a meaningful tradeoff: it keeps the controller simple (no exception throwing, no NestJS `HttpException` imports) and the client always gets a parseable JSON body. However, it breaks the HTTP contract — clients, proxies, monitoring tools, and API gateways all treat `200` as success, so a `{ error: "Order with id X not found" }` payload will not trigger alerts, will not be retried by standard retry middleware, and will confuse any consumer that deserializes the response directly into an `Order` type (TypeScript type assertions like `as Order[]` in `App.tsx` become dangerous).

For a real-world POS API I would make three changes. First, throw `NotFoundException` from `@nestjs/common` when `findOne` returns `undefined` — NestJS maps this to a proper `404` with a standard `{ statusCode, message, error }` body. Second, add a global `ValidationPipe` and a consistent exception filter so every error, including unhandled ones, returns the same shape (e.g., `{ statusCode, message, timestamp, path }`). This lets frontend and monitoring code handle errors uniformly. Third, for the `GET /api/orders` list endpoint I would add explicit `400 Bad Request` responses for invalid query parameters (e.g., malformed date filters or negative page numbers) rather than silently ignoring them, since silent failures are hard to debug in a multi-terminal POS environment.

## 3.

The current pattern in `App.tsx` — a `useEffect` with an inline `async` function, raw `fetch`, manual `loading`/`error` state, and a hardcoded `http://localhost:3001` URL — is fine for a two-component demo but becomes hard to maintain as the dashboard grows. The problems compound quickly: each new API call needs its own copy of the loading/error/data triple, the URL is scattered across files rather than centralised, cancellation on unmount is not handled (missing `AbortController`), and there is no caching so navigating away and back re-fetches everything.

I would restructure the frontend in three layers. First, centralise all HTTP calls in a dedicated `api/` module (e.g., `api/orders.ts`) that exports typed async functions like `fetchOrders()` and `fetchOrder(id)`. The base URL comes from an environment variable (`REACT_APP_API_URL`), making environment switching trivial. Second, replace raw `useEffect`+`fetch` with a data-fetching library — React Query (TanStack Query) is the right choice here because it handles caching, background refetching, pagination, and loading/error states declaratively. A component like `OrdersList` would call `useQuery(['orders'], fetchOrders)` and the repetitive state management disappears. Third, for filters and pagination I would lift query parameters into the URL using `react-router` search params so that the dashboard state is bookmarkable and shareable — important for a POS where staff might share a filtered view of "ready" orders with a colleague.

## 4.

The current `Order` and `Garment` types in `orders.service.ts` are minimal by design but miss several fields that are essential even in a small real-world laundry operation.

On the `Order` level: there is no `status` field on the order itself — the overall order status must currently be inferred by inspecting every garment's status, which is fragile and expensive. There is no `dueDate` or `promisedAt` field, which is the most customer-visible piece of information in a dry-cleaning workflow. There is no `totalAmount`, `paidAmount`, or `paymentMethod`, making billing impossible. There is no reference to a customer entity beyond a plain `customerName` string — repeat customers cannot be tracked, and phone/email for notifications are missing.

On the `Garment` level: there is no `serviceType` (e.g., dry-clean, wash-and-fold, press-only), which determines pricing and processing time. There is no `price` field, so line-item billing cannot be computed. There is no `notes` field for special instructions (e.g., "treat stain on collar"), which is operationally critical. The `GarmentStatus` union is missing a `cancelled` state, which means refund and partial-order workflows cannot be represented.

Edge cases the current model cannot express: an order where some garments are `delivered` and others are `in_cleaning` (partial fulfillment), prepaid packages where a customer has credits that must be drawn down across multiple orders, and rush orders that should surface at the top of the processing queue. I would evolve the model by adding `status` and `dueDate` to `Order`, making `Customer` a first-class entity with a foreign key, adding `serviceType` and `price` to `Garment`, and expanding `GarmentStatus` to include `cancelled`.

## 5.

The `OrdersService` methods `findAll` and `findOne` are simple enough that AI generation adds little risk — the logic is a direct array lookup and the types are right there in the same file. The higher risk sits at the boundary between the generated code and its callers. An AI tool might generate a `findOne` that returns `null` instead of `undefined`, subtly breaking the strict equality check `if (!order)` in `orders.controller.ts` (though both are falsy, the TypeScript return type `Order | undefined` would be wrong). Similarly, the current controller returns a `200` with `{ error: string }` on not-found — a pattern an AI might replicate everywhere because it saw it once in context, silently propagating a design flaw across the codebase.

On the frontend, the risk is in type assertions. `App.tsx` casts the fetch response with `as Order[]` without any runtime validation. If an AI-generated backend change adds or renames a field (e.g., renames `customerName` to `customer`), the TypeScript compiler passes because the `as` cast bypasses type checking, and the bug only appears at runtime in the UI.

The review practices I would apply before shipping: (1) treat AI output as a first draft — always read every line before committing, paying special attention to error paths and type boundaries; (2) add runtime schema validation with a library like `zod` on both the API response in the frontend and the incoming request bodies in the backend, so type mismatches surface immediately as clear errors rather than silent rendering bugs; (3) write at least one integration test per endpoint that covers the not-found case — the current codebase has no tests, which means AI-generated logic in `findOne` has never been exercised beyond manual `curl` calls; (4) check that generated code respects the existing error-handling conventions of the codebase, rather than introducing a second inconsistent pattern.

## 6.

The current architecture is purely request-response REST, so the dashboard has no way to learn about garment status changes without polling. For a "garment board" that updates when a garment moves to `ready`, the two realistic approaches are polling and Server-Sent Events (SSE), with WebSockets as a third option for bidirectional needs.

**Short-poll** (simplest): have the React client call `GET /api/orders` every 5–10 seconds. This requires zero backend changes and works through every firewall and proxy. The tradeoff is wasted bandwidth and latency up to one poll interval — acceptable for a back-office dashboard but not for a customer-facing "your order is ready" screen.

**Server-Sent Events** (best fit here): add a `GET /api/orders/events` endpoint in NestJS that keeps the HTTP connection open and pushes `text/event-stream` messages whenever an order or garment status changes. The frontend subscribes with the browser's native `EventSource` API. This is strictly unidirectional (server → client), which is exactly what a status board needs, and it works over standard HTTP/1.1 without the upgrade complexity of WebSockets. The tradeoff is that SSE connections count against the browser's per-origin connection limit (6 for HTTP/1.1), which matters if multiple dashboard tabs are open simultaneously — HTTP/2 eliminates this limit.

**WebSockets** (overkill for read-only boards, right choice if staff also submit status updates from the UI): NestJS has first-class `@WebSocketGateway` support. The added complexity of connection management, reconnect logic, and room/namespace design is only justified if the real-time channel needs to carry writes as well as reads.

Regardless of the transport chosen, the backend needs a mechanism to detect state changes and broadcast them. With the current in-memory array this is straightforward — an `updateGarmentStatus` method in `OrdersService` can emit an event via NestJS's `EventEmitter2` module after mutating the array, and the SSE or WebSocket gateway subscribes to that event. At production scale with a database, I would use PostgreSQL's `LISTEN`/`NOTIFY` or a Redis pub/sub channel so that all horizontally-scaled server instances receive the same change events and broadcast to their own connected clients.
